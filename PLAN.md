# Plan: package the CCPM scheduler as a library + agent-friendly CLI

Status: agreed 2026-07-12. Phases are ordered; each lands as its own reviewed change.

## Goal

Turn the `skills/ccpm-scheduler/scripts/` into a proper Python package that serves
three consumers from one core:

```
                 ┌──────────────────┐
                 │  ccpm (library)   │  pure core: typed model, validate,
                 │  src-layout pkg   │  build, check, plot — importable API
                 └───────┬──────────┘
        ┌────────────────┼────────────────────┐
  ccpm-scheduler CLI   our-planner          Claude skill
  (console script,    (imports library     (SKILL.md tells agents
   agent-friendly)     directly, in-proc)   to run the CLI)
```

## Decisions (agreed)

- **Package home**: dedicated repo (this one, `~/workspace/ccpm-scheduler`),
  publishing to PyPI as `ccpm-scheduler` (verified free on 2026-07-12; note the
  `ccpm` distribution is taken, hence the import name `ccpm_scheduler`).
  Until the first PyPI release, our-planner and the skill repo consume it as a
  path/git dependency.
- **CLI name**: `ccpm-scheduler` (console entry point and distribution name).
- **Fractional resources (v1)**: reject with a clear, actionable validation error.
  our-planner allows fractional allocations (task uses 0.5 of a resource) and
  fractional per-day capacity; the integer leveling algorithm stays as-is for v1
  and float support is a separate, later phase with its own determinism testing.
- **Duration naming**: `realistic_duration` / `optimal_duration` (matches the
  our-planner task model; legacy `duration_safe`/`duration_aggressive` accepted
  as aliases — done, commit 61a9598).

## Why the fit is good

our-planner (`/home/rnwolf/workspace/our-planner`) already:
- stores `realistic_duration` / `optimal_duration` per task
- uses the identical dependency-link notation (`A`, `A:SS+2`, `A:FF`), with
  `PB`/`FB` link types explicitly reserved for CCPM buffer rows it does not
  yet generate (`src/model/dependency_notation.py`)
- models resources as per-day capacity vectors (run-length-encodes cleanly into
  the `[from, to)` calendar windows the builder already understands)
- has projects, chains (with `is_critical`), task `type`, and day-offset
  positioning that map directly onto the schedule output

Gaps: fractional allocations/capacity (see decision above); int task ids vs
string ids (trivial).

## Phase 1 — Extract the library (behavior-preserving)

**Done (Phase 1a, 2026-07-12)**: repo scaffolded; the four skill scripts moved
verbatim to `src/ccpm_scheduler/{validate,build,check,plot}.py` (PEP 723
headers stripped, `python -m` entry points kept); golden tests assert
byte-identical `schedule.csv`/`summary.md` against outputs generated by the
original scripts for 5 reference projects, plus legacy-alias, bad-network,
and PNG smoke tests (18 tests). License: MIT OR Apache-2.0 (dual).

**Done (Phase 1b, 2026-07-12, v0.2.0)** — the typed model and library API:
  - `ccpm_scheduler/model.py` — dataclasses: `Task`, `Resource`, `CalendarWindow`,
    `Link`, `Schedule`/`ScheduleRow`, `Issue`/`ValidationReport`,
    `BuildResult`/`BuildStats`, `CcpmError`. Every validation issue carries a
    machine code (`E_CYCLE`, `E_NO_RESOURCE`, `E_FRACTIONAL_*`, …), severity,
    offending ids, and a human message — this is what the GUI renders inline.
  - `validate_network(net)`, `build_schedule(net, title)`,
    `check_schedule(schedule, net)`, `plot_schedule(schedule, out, ...)` —
    the four stages as library functions; the `python -m` CLIs are thin
    wrappers with unchanged output text and exit codes.
  - `ccpm_scheduler/io.py` — CSV adapter (existing contract, legacy aliases)
    + JSON exchange format: structured predecessor links (`{"id","type","lag"}`),
    numeric ids coerced to strings, per-resource allocation maps (fractional
    allocations/capacities rejected with precise E_FRACTIONAL_* errors).
  - matplotlib imports lazily — library consumers don't pay for the plotter.
  - 38 tests: the original golden suite (byte-identical outputs, unchanged)
    plus API pipeline, JSON-roundtrip-to-golden, coded-validation, tampering,
    and our-planner-shape tests.

## Phase 2 — The agent-friendly CLI

**Done (2026-07-12, v0.3.0)**: `ccpm-scheduler` console script (also
`python -m ccpm_scheduler`) with subcommands mirroring the workflow:
`validate`, `build`, `check`, `plot`, `schema` (emits JSON Schemas for the
network input, schedule output, and validation report so an agent can
self-serve the contracts). Design rules implemented:

- `--json`: machine-readable results on stdout (validation issues as coded
  records; build emits stats + file paths + the full schedule)
- exit codes: 0 = ok, 1 = validation/schedule errors (report still emitted),
  2 = usage error — agents branch on codes, not prose
- `build` validates first: broken networks yield the coded issue report and
  no output files
- network input as CSV files or a single JSON document (path or `-` = stdin)
- no interactive prompts, deterministic output, no color
- error messages say how to fix, not just what's wrong
- 21 CLI contract tests (exit codes, --json shapes, determinism)

## Phase 4 — our-planner integration

**Done (2026-07-12, our-planner branch `stage-16-ccpm-roundtrip`, commit
3fc7a1e)** — implemented as our-planner's own planned Stage 16 ("CCPM round
trip"): `CcpmOperations` (`src/operations/ccpm_operations.py`) with this
package as a git dependency, two File-menu items:

1. **Export CCPM Network...** — chosen project → `tasks.csv`/`resources.csv`/
   `calendar.csv` in this package's input format (per-day capacity vectors
   run-length-encoded into `[from, to)` windows).
2. **Schedule with CCPM...** — in-process: `network_from_json` (structured
   links, allocation maps) → `validate_network` (coded issues in a dialog on
   failure) → `build_schedule` → `check_schedule` → schedule imported as a
   NEW project `<source> (CCPM)` beside the original via the pre-existing
   `import_ccpm_schedule` machinery — buffers become `project_buffer`/
   `feeding_buffer` tasks with `PB`/`FB` links, chains map onto the app's
   chain model, and the shared resource pool is reused by name.

Applying the CCPM schedule to a *new* project (side-by-side comparison)
rather than mutating the source project follows our-planner's own Stage 16
design; complete tasks are excluded, buffers never exported, and fractional
allocations surface this package's E_FRACTIONAL_ALLOCATION error. 11 tests in
our-planner mirror the worked example (CC 30d, PB 15, promise day 45).
